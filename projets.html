Sécurisation des accès réseau filaire — 802.1X, RADIUS & VLAN dynamiques

🏢 GSB 🌐 Réseau 🔒 Sécurité 📅 Fév – Mars 2026

Dans le cadre de la politique de sécurité de GSB (Galaxy Swiss Bourdin), toute personne connectant un ordinateur sur une prise Ethernet accédait librement au réseau interne. La solution déployée impose une authentification 802.1X via un serveur RADIUS (NPS) et attribue automatiquement un VLAN selon le profil utilisateur, isolant les non-autorisés dans un réseau invité.

🖥️ Infrastructure déployée

ComposantRôleAdresse IP
Switch HP ProCurve 2650Authentification des ports 802.1X—
Serveur AD (VM)AD DS · DNS · NPS (RADIUS) · AD CS192.168.1.10 /24
Client AutoriséPoste Windows domaine mars.local192.168.1.20 /24
Client RefuséMachine hors-domaine → VLAN invité192.168.1.30 /24

🔀 Segmentation VLAN

VLAN 10 — Service Administratif VLAN 20 — Service Commercial VLAN 99 — Réseau Invité

⚙️ Mise en œuvre

  • Création des groupes AD Administratif et Commercial
  • Installation et enregistrement du rôle NPS dans Active Directory
  • Ajout du switch comme client RADIUS avec secret partagé
  • Déploiement d'une Autorité de Certification (AD CS) et certificat serveur
  • Activation 802.1X sur les ports du switch HP ProCurve — VLAN invité pour refus
  • Authentification MAC pour les imprimantes non compatibles 802.1X
  • Configuration PEAP / EAP-MSCHAPv2 sur les postes clients Windows
802.1X RADIUS / NPS Active Directory AD CS / PKI PEAP / EAP-MSCHAPv2 VLAN dynamiques HP ProCurve Windows Server

Protection anti-brute force avec Fail2Ban & centralisation des journaux rsyslog

🏢 GSB 🐧 Linux 🔒 Sécurité 📅 Fév – Mars 2026

Un audit interne de GSB a révélé l'absence de mécanisme de blocage automatique face aux tentatives SSH répétées et aux attaques web automatisées. La solution déployée associe Fail2Ban (bannissement d'IP) à une infrastructure rsyslog centralisée pour une supervision complète des événements de sécurité.

🖥️ Environnement virtualisé

MachineRôle & ServicesIP
VM1 — Serveur WebApache · SSH · Fail2Ban · rsyslog192.168.1.10 /24
VM2 — Serveur FichiersSamba · SSH · Fail2Ban · rsyslog192.168.1.20 /24
VM3 — ClientAccès web, SSH, partage de fichiers192.168.1.30 /24
VM4 — AttaquantSimulation de brute force192.168.1.40 /24
VM5 — Supervisionrsyslog — centralisation des journaux192.168.1.50 /24

🔏 Politique de sécurité

  • SSH (port 22) — accès réseau interne uniquement avec identifiants valides
  • HTTP (port 80) — accès libre en lecture via Apache (réseau local)
  • Samba (port 445) — accès restreint au client autorisé
  • rsyslog (port 514 UDP) — transmission automatique des journaux vers VM5
  • Fail2Ban — bannissement automatique des IP après tentatives répétées sur SSH / Apache

⚙️ Mise en œuvre

  • Installation et configuration de Fail2Ban sur VM1 et VM2
  • Création de jails SSH et Apache (seuil de tentatives, durée de ban)
  • Configuration de rsyslog pour transmission UDP vers VM5 (serveur de supervision)
  • Mise en place du serveur de supervision (VM5) pour la centralisation des logs
  • Configuration de Samba pour le partage sécurisé de fichiers
Fail2Ban rsyslog Apache Samba SSH Ubuntu Server iptables Supervision

✅ Tests réalisés

Test 1 — Connexion utilisateur

VM3 → SSH réussi sur VM1 & VM2, accès web et partage Samba fonctionnels.

Test 2 — Brute force

VM4 lance une attaque SSH → IP automatiquement bannie par Fail2Ban sur VM1 et VM2.

Test 3 — Journaux

VM5 reçoit et enregistre correctement les logs des deux serveurs en temps réel.
